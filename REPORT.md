# Análisis Técnico de la Vulnerabilidad "Copy Fail" (CVE-2026-31431)

### Mecanismo del Bug Raíz y Corrupción de Memoria
La vulnerabilidad "Copy Fail" proviene del subsistema criptográfico del Kernel de Linux, en particular dentro de la interfaz de sockets de algoritmos de usuario (`AF_ALG`), situada en el archivo fuente `crypto/algif_aead.c`, específicamente en la función de procesamiento de datos decodificados. El error fundamental es un fallo de cálculo lógico en la manipulación de buffers de memoria durante procesos de descifrado autenticado con datos relacionados (AEAD). Al manejar una solicitud maliciosa, el kernel evalúa erróneamente el tamaño total necesario para los datos de destino, confundiendo el desplazamiento relativo del búfer fuente con el del de destino.

Debido a este desajuste, el kernel ejecuta una operación de escritura fuera de los límites en la dirección del buffer `dst[assoclen + cryptlen]`. Esta escritura es sumamente arriesgada ya que el destino señalado no dirige a una memoria volátil aislada del proceso de usuario, sino que reemplaza directamente páginas físicas de memoria compartida que pertenecen al espacio del núcleo. Al modificar este índice de manera aleatoria, un atacante puede insertar bytes de código dañino directamente en estructuras de datos cercanas del sistema operativo, alterando el flujo de ejecución del núcleo.

### Persistencia en Memoria y Conceptos del Sistema de Archivos
El exploit se caracteriza por su sigiloso comportamiento o encubrimiento estratégico (*stealthy*), ya que altera únicamente la memoria RAM y nunca efectúa una modificación física en el archivo binario guardado en el disco duro. Para entender este fenómeno, es fundamental vincular el diseño del núcleo con los conceptos de inodos y el sistema de almacenamiento. En Linux, un archivo almacenado en disco se representa mediante un inodo que determina sus metadatos y permisos fijos, como los bits `setuid` y las limitaciones de lectura/escritura (`chmod`). Cuando un proceso pide ejecutar un binario del sistema como `/bin/su`, el sistema operativo no accede al disco directamente por razones de rendimiento; en cambio, transfiere el contenido del archivo al *Page Cache* (la memoria RAM del sistema de archivos).
El exploit combina llamadas al sistema como `splice` para integrar el *Page Cache* del binario `su` en el socket criptográfico que es vulnerable. Al provocar el desbordamiento en `algif_aead.c`, el núcleo daña la copia en memoria RAM de `/bin/su`, substituyendo sus rutinas de verificación de contraseña por instrucciones que permiten acceso directo. Debido a que los permisos del inodo original en el disco indican que `/bin/su` tiene el bit `setuid` habilitado, el proceso corre con la identidad del dueño del archivo (root, UID 0). El sistema operativo es manipulado para conceder una terminal de superusuario auténtica utilizando una memoria RAM modificada, mientras que el inodo y el archivo real en el disco se mantienen completamente intactos y limpios frente a cualquier revisión forense convencional.


### Lecciones sobre la Complejidad del Software
Este laboratorio proporciona una lección fundamental en seguridad informática: la peligrosidad de la complejidad sistémica. Individualmente, los subsistemas de criptografía (`AF_ALG`), la optimización de transferencia de datos de memoria a memoria (`splice`) y el mecanismo de aceleración de lectura (*Page Cache*) son componentes razonables, eficientes y bien diseñados bajo sus propios supuestos de aislamiento. Sin embargo, la interacción imprevista entre estos tres paradigmas legítimos crea una superficie de ataque catastrófica. Esto demuestra que la seguridad no es una propiedad estática de componentes individuales, sino una propiedad emergente del sistema completo, evidenciando que modificaciones lógicas pequeñas o parches de optimización que parecen inofensivos de forma aislada pueden colisionar destructivamente y comprometer la integridad absoluta del sistema operativo.
